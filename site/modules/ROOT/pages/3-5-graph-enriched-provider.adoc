= Graph-Enriched Retrieval
:branch: main

In the previous lesson, you used vector search to find relevant movies by semantic similarity. The search returned matching plot text, but your graph contains far more than text. Each movie is connected to genres, actors, and directors through typed relationships. Basic search misses all of that structure.

Graph-enriched retrieval adds a Cypher traversal step after the index search. The provider finds matching movies, then walks the graph to collect genres, cast, and directors for each match. The agent receives both the relevant text and the structured metadata.

== The Limitation of Basic Search

The vector search from the previous lesson returns matching text and a similarity score for each result. If you search for "time travel," you get plots that mention time travel concepts. But the result doesn't include the movie's title, release year, genres, or who starred in it.

That metadata lives in the graph: `Movie` nodes connect to `Genre` nodes via `IN_GENRE`, to actors via `ACTED_IN`, and to directors via `DIRECTED`. Graph enrichment traverses these relationships after the initial search.

== Define a Retrieval Query

The `retrieval_query` parameter accepts a Cypher query that runs after the index search. It receives two variables:

* **`node`** -- The matched `Movie` node from the index
* **`score`** -- The relevance score from the search

The query traverses outward from the matched node:

[source,cypher]
----
// Receives :param node and :param score from the index search
MATCH (node)-[:IN_GENRE]->(g:Genre)
WITH node, score, collect(DISTINCT g.name) AS genres
OPTIONAL MATCH (p:Person)-[:ACTED_IN]->(node)
WITH node, score, genres, collect(DISTINCT p.name)[0..5] AS actors
OPTIONAL MATCH (d:Person)-[:DIRECTED]->(node)
WITH node, score, genres, actors, collect(DISTINCT d.name) AS directors
WHERE score IS NOT NULL
RETURN
    node.plot AS text,
    score,
    node.title AS title,
    node.released AS released,
    genres,
    actors,
    directors
ORDER BY score DESC
----

This query starts from each matched movie and collects:

* Genres via `Movie` -> `IN_GENRE` -> `Genre`
* Up to 5 actors via `Person` -> `ACTED_IN` -> `Movie`
* Directors via `Person` -> `DIRECTED` -> `Movie`

The result includes plot text, similarity score, movie title, release year, genre names, actor names, and director names. Far richer than plot text alone.

What is the purpose of the `retrieval_query` parameter in `Neo4jContextProvider`?

* [ ] It replaces the vector search entirely with a Cypher query

[TIP,role=solution]
.Solution
====
The correct answer is: **It traverses graph relationships after index search to include structured context from related nodes.**

The `retrieval_query` is a Cypher query that runs after the vector (or fulltext) search. It receives the matched `node` and `score` from the index search, then traverses relationships to collect additional data from connected nodes. In the movie graph, this means following `IN_GENRE`, `ACTED_IN`, and `DIRECTED` relationships to include genre names, actor names, and directors alongside the matched plot text.

== Lesson Summary

Adding a `retrieval_query` to `Neo4jContextProvider` combines any search mode with graph traversal. The Cypher query receives matched `node` and `score` variables from the index search, then traverses relationships to collect genres, actors, and directors. The agent gets structured metadata alongside plot text, producing more specific and grounded responses.
