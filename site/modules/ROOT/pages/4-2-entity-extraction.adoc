= Entity Extraction Pipeline

In the previous lesson, you learned that long-term memory stores entities, preferences, and facts extracted from conversations. But how does the extraction happen? The `neo4j-agent-memory` package uses a multi-stage pipeline that combines fast pattern matching with accurate model-based extraction to pull structured entities from unstructured conversation text.

== The Extraction Pipeline

Entity extraction runs as a three-stage pipeline. Each stage has different strengths, and the results are merged to maximize coverage and accuracy:

1. **spaCy** -- Fast, rule-based named entity recognition. Identifies common entity types (people, organizations, locations) using linguistic patterns. Runs locally with no API calls.
2. **GLiNER** -- A lightweight model specialized for entity extraction. More accurate than spaCy for domain-specific entities. Supports POLE+O schema types out of the box.
3. **LLM fallback** -- Uses a language model to extract entities that the first two stages missed. Most accurate but slowest and most expensive.

Each stage runs in sequence. If spaCy catches "Christopher Nolan" as a PERSON, GLiNER confirms it and may add a confidence score, and the LLM stage can skip entities already found.

== Configuring Extraction

The extraction behavior is configured through `MemorySettings`:

[source,python]
----
from neo4j_agent_memory import MemorySettings

settings = MemorySettings(
    neo4j={
        "uri": os.environ["NEO4J_URI"],
        "username": os.environ["NEO4J_USERNAME"],
        "password": SecretStr(os.environ["NEO4J_PASSWORD"]),
    },
    extraction={
        "extractor_type": "pipeline",
        "enable_spacy": True,
        "enable_gliner": True,
        "enable_llm_fallback": True,
        "confidence_threshold": 0.5,
        "entity_types": [
            "PERSON", "ORGANIZATION", "LOCATION", "EVENT", "OBJECT"
        ],
    },
)
----

Key parameters:

* **`extractor_type`** -- Set to `"pipeline"` for the multi-stage approach. Can also be `"llm"` for LLM-only extraction.
* **`enable_spacy`**, **`enable_gliner`**, **`enable_llm_fallback`** -- Toggle individual pipeline stages.
* **`confidence_threshold`** -- Minimum confidence score (0.0 to 1.0) for an entity to be stored. Lower values capture more entities but with more noise.
* **`entity_types`** -- The POLE+O types to extract. Limits extraction to these categories.

== Merge Strategies

When multiple stages find the same entity, the pipeline needs to decide which result to keep. The `merge_strategy` parameter controls this:

* **`confidence`** -- Keeps the result with the highest confidence score per entity. The default and usually the best choice.
* **`union`** -- Keeps all unique entities from all stages. Maximizes coverage but may include duplicates.
* **`intersection`** -- Only keeps entities found by multiple stages. Maximizes precision but may miss entities only one stage caught.
* **`cascade`** -- Uses results from the first stage, fills gaps with subsequent stages. Fast because later stages only process what earlier ones missed.

[source,python]
----
settings = MemorySettings(
    # ...
    extraction={
        "extractor_type": "pipeline",
        "merge_strategy": "confidence",
    },
)
----

== Entity Deduplication

As conversations continue, the same entity may be mentioned in different ways: "Christopher Nolan," "Nolan," "the director of Inception." The package includes automatic deduplication using embedding similarity:

* Entities above the **auto-merge threshold** (default 0.95) are merged automatically
* Entities between the **flag threshold** (default 0.85) and auto-merge are flagged for review
* Entities below the flag threshold are treated as distinct

This deduplication is configured through `resolution` settings:

[source,python]
----
settings = MemorySettings(
    # ...
    resolution={
        "strategy": "composite",
        "exact_threshold": 1.0,
        "fuzzy_threshold": 0.85,
        "semantic_threshold": 0.8,
    },
)
----

== How Entities Map to Neo4j

Extracted entities become nodes in the Neo4j graph. The POLE+O type and subtype become Neo4j labels in PascalCase:

[cols="1,1,2"]
|===
| Entity Type | Subtype | Neo4j Labels

| PERSON
| INDIVIDUAL
| `:Entity:Person:Individual`

| OBJECT
| VEHICLE
| `:Entity:Object:Vehicle`

| ORGANIZATION
| COMPANY
| `:Entity:Organization:Company`

| LOCATION
| CITY
| `:Entity:Location:City`

| EVENT
| MEETING
| `:Entity:Event:Meeting`
|===

Each entity node stores the name, type, subtype, description, confidence score, and a vector embedding. Entity nodes link back to the conversation messages where they were mentioned, creating a connected knowledge structure the agent can traverse.

== Automatic vs Manual Extraction

When `extract_entities=True` is set on `Neo4jMicrosoftMemory`, entity extraction runs automatically in `after_run()` after every agent response. By default, this runs asynchronously in the background so it does not slow down the agent's response time.

You can also extract entities manually:

[source,python]
----
entity = await memory_client.long_term.add_entity(
    name="Inception",
    entity_type="OBJECT",
    description="2010 science fiction film directed by Christopher Nolan",
    confidence=0.95,
)
----

Manual extraction is useful when you want to pre-populate the memory graph with known entities before conversations begin.

The `neo4j-agent-memory` extraction pipeline uses three stages (spaCy, GLiNER, LLM) to extract structured entities from conversation text.
