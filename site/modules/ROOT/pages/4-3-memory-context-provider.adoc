= Memory as a Context Provider
:branch: main

TIP: Complete this lesson hands-on by opening `labs/lab-7-memory-provider.ipynb` in your Codespace.

The `Neo4jContextProvider` from Module 3 injected knowledge graph data before every agent turn. The memory context provider does the same thing, but for memories the agent has accumulated during conversations. Before each turn, it retrieves relevant past messages, extracted entities, and user preferences. After each turn, it stores new messages and extracts entities using the pipeline from the previous lesson.

The interface is `Neo4jMicrosoftMemory`, which wraps the memory client into a context provider and chat message store.

== Configure Memory Settings

Connect to Neo4j and create a memory client:

[source,python]
----
# Configure memory settings with Neo4j connection
settings = MemorySettings(
    neo4j={
        "uri": os.environ["NEO4J_URI"],
        "username": os.environ["NEO4J_USERNAME"],
        "password": SecretStr(os.environ["NEO4J_PASSWORD"]),
    },
)

----

NOTE: `MemorySettings` also supports automatic environment variable loading with the `NAM_` prefix (e.g., `NAM_NEO4J__URI`). The explicit configuration shown here reuses the same `NEO4J_URI`, `NEO4J_USERNAME`, and `NEO4J_PASSWORD` environment variables used by the Neo4j context provider lessons.

Use `MemoryClient` as an async context manager to open the connection:

[source,python]
----
async with MemoryClient(settings) as memory_client:
    # memory_client is now connected to Neo4j
    ...
----

The `async with` block automatically connects on entry and closes the connection on exit. The `MemoryClient` exposes the short-term, long-term, and reasoning memory APIs.

== Create the Unified Memory

`Neo4jMicrosoftMemory.from_memory_client()` takes the client and returns an object that provides both a context provider and a chat message store:

[source,python]
----
async def main():
    async with MemoryClient(settings) as memory_client:
        # Create unified memory interface
        session_id = "movie-chat-session"

        memory = Neo4jMicrosoftMemory.from_memory_client(
            memory_client=memory_client,
            session_id=session_id,
            include_short_term=True,
            include_long_term=True,
            include_reasoning=True,
            extract_entities=True,
        )
        # end::memory[]

        # tag::agent[]
        client = OpenAIResponsesClient()

        agent = client.as_agent(
            name="movie-memory-agent",
            instructions=(
                "You are a movie recommendation assistant with persistent "
                "memory. You remember what users have told you about their "
                "preferences, which movies you have discussed, and what "
                "you recommended in past sessions."
            ),
            context_providers=[memory.context_provider],
        )

        session = agent.create_session()
        # end::agent[]

        # tag::run[]
        queries = [
            "I really enjoy sci-fi movies, especially ones about time travel.",
            "What did I say my favorite genre was?",
            "Can you recommend something I might like?",
        ]

        for query in queries:
            print(f"\nUser: {query}")
            response = await agent.run(query, session=session)
            print(f"Assistant: {response.text}")
        # end::run[]

        # tag::search[]
        results = await memory.search_memory(
            query="sci-fi movies",
            include_messages=True,
            include_entities=True,
            include_preferences=True,
            limit=5,
        )

        print("Messages:", results.get("messages", []))
        print("Entities:", results.get("entities", []))
        print("Preferences:", results.get("preferences", []))
        # end::search[]

asyncio.run(main())

----

The parameters control what gets injected and extracted:

* **`session_id`** -- scopes the conversation. Messages and entities are grouped by session.
* **`include_short_term`** -- injects recent and semantically relevant past messages before each turn
* **`include_long_term`** -- injects extracted entities and preferences before each turn
* **`include_reasoning`** -- injects similar past reasoning traces before each turn
* **`extract_entities`** -- pulls entities (people, movies, genres) out of messages after each turn

== The Memory Lifecycle

The provider follows the same `before_run()` / `after_run()` lifecycle from Module 3, applied to memory retrieval and storage.

**`before_run()`** searches across memory types and injects what it finds as instructions. For a query about sci-fi recommendations, it might inject recent conversation messages mentioning sci-fi, a stored preference for Christopher Nolan films, and a past reasoning trace where the agent successfully recommended Interstellar.

**`after_run()`** saves the new messages (both the user's input and the agent's response) to short-term memory with vector embeddings, then runs entity extraction. If the user mentioned "Inception," the extractor creates an entity node for that movie and links it to the conversation.

None of this requires the agent to call a tool. The framework runs the provider automatically on every turn.

Walk through what happens on each turn:

1. **Turn 1** -- The user states a preference. `after_run()` saves the message, generates a vector embedding, and extracts "sci-fi" and "time travel" as entities linked to the session.
2. **Turn 2** -- The user asks what they said. `before_run()` retrieves the previous message from short-term memory (via semantic search on "favorite genre") and injects it as context. The agent answers accurately because the memory is in its context window.
3. **Turn 3** -- The agent has both the conversation history and the extracted preference available. It can make a recommendation grounded in what the user actually said, not a generic suggestion.

== Lesson Summary

`Neo4jMicrosoftMemory` provides a context provider that retrieves relevant memories in `before_run()` and stores new messages and extracted entities in `after_run()`. The agent gets persistent memory across turns and sessions without calling any tools.
